## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Timezone off-by-one in booking date labels**
- **Found during:** Task 3
- **Issue:** Formatting the `Date.UTC(...)`-anchored day value with `timeZone: "America/New_York"` (as literally specified in the plan's interfaces snippet) shifts the displayed weekday/month/day back by one calendar day versus the `iso` value, since NY is behind UTC and the anchor instant is UTC midnight.
- **Fix:** Format with `timeZone: "UTC"` instead for both `SHORT_DATE_FORMATTER` and `LONG_DATE_FORMATTER`, keeping the "today" anchor's NY-zone lookup unchanged. Verified with a standalone Node script showing `iso: 2026-07-21` / NY-format `Mon, Jul 20` (wrong) vs. UTC-format `Tue, Jul 21` (correct, matches iso).
- **Files modified:** `src/lib/booking.ts`

## User Setup Required

**External services require manual configuration before booking notifications work in production.** See `.env.local.example` at the repo root for step-by-step setup:
- Message @BotFather on Telegram to create a bot and obtain `TELEGRAM_BOT_TOKEN`.
- Message the new bot once, then visit `https://api.telegram.org/bot<TOKEN>/getUpdates` (or use a helper bot like @userinfobot) to read back the numeric `TELEGRAM_CHAT_ID`.
- Set both values in the deployment's environment (e.g., Netlify site environment variables) and in a local `.env.local` (untracked) for local development.
- Until both env vars are set, `/api/book` returns a 500 with a generic "Booking is temporarily unavailable" message to visitors and logs the specific missing-var reason server-side — it fails safely, not silently.

The Formspree endpoint (`formspree.io/f/mjgnobjw`) is already created per the plan's context and required no additional setup.
