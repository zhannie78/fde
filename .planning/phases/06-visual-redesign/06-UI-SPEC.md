| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 regular, JetBrains Mono | 1.75 |
| Label | 14px (`text-sm`), uppercase, `letter-spacing: 0.03–0.06em` | 700 bold, JetBrains Mono | 1.4 |
| Heading | `clamp(1.75rem, 3.2vw, 2.75rem)` (28–44px), `letter-spacing: -0.02em` | 700 bold, JetBrains Mono | 1.2 |
| Display | `clamp(2.75rem, 6.4vw, 5rem)` (44–80px), `letter-spacing: -0.035em` | 700 bold, JetBrains Mono | 1.0 |

Weight count: exactly 2 (400 / 700) — matches sketch finding's explicit "only 2 font weights used... do not introduce more" gate.

**Display is reserved for the Hero H1 only** (unchanged rule from Phase 5, still correct under the new type system) — every other heading uses the Heading role, including the About page's name/title.

Apply uniformly; do not mix 1.6 and 1.75 across sections.

**Inline emphasis pattern (replaces any inline accent-coloring of emphasis phrases):** for phrases like "save time" / "increase profit" inside body copy, do NOT recolor text into the accent — keep near-black + bold (700), add `box-shadow: inset 0 -2px 0 0 var(--accent)` as an underline rule, plus `background: linear-gradient(transparent 72%, var(--accent-soft) 72%)` as a highlighter-style tint. This is a locked CSS pattern, not a suggestion — reads as "deliberate highlighter," a flat accent color on inline text "reads like a broken link" (sketch finding, corrected once already during sketching).

## Motion & Performance Contract (DSGN-02 / DSGN-03 / DSGN-04)

### Stack decision: GSAP + ScrollTrigger for scroll-driven narrative, `motion` retained for discrete micro-interactions

Resolves the `STATE.md` blocker ("Motion vs. GSAP+ScrollTrigger... confirm choice when Phase 6 is planned"):

- **GSAP (`gsap` + `@gsap/react`'s `useGSAP` hook) + `ScrollTrigger`** — NEW dependency, install for this phase. Confirmed 100% free as of the Webflow acquisition (Oct 2024), including `ScrollTrigger` and every formerly-Club plugin — no license cost, consistent with the project's lean/free-tier budget constraint. Use exclusively for scroll-position-tied choreography (the 5-part narrative, DSGN-02). `useGSAP`'s automatic context cleanup (kills animations/ScrollTriggers, reverts DOM on unmount) is required — do not call `gsap.to`/`ScrollTrigger.create` directly inside a bare `useEffect`.
- **`motion` (already installed, already used in `glow.tsx`)** — KEEP for what it already does: discrete, non-scroll-linked micro-interactions (button hover/tap spring, the breathing glow-wrap pulse). Do not migrate `glow.tsx` to GSAP — no functional benefit, adds rework risk. Do not use `motion` for anything scroll-position-tied — that's GSAP's job exclusively, to avoid two libraries fighting over the same scroll listener.
- **Bundle discipline:** `gsap` core (~28kb gzip) + `ScrollTrigger` (~10kb gzip) must be dynamically imported inside a client-only wrapper (`next/dynamic(() => import(...), { ssr: false })` or a `"use client"` component that imports GSAP inside `useGSAP`'s effect body, not at module top-level of anything server-rendered). Never import GSAP in `layout.tsx` or any Server Component.

### Section-by-section choreography (executor build list, DSGN-02)

Maps the 5 conceptual narrative acts onto Phase 5's 7 locked components. Do not add new sections to achieve this — apply these effects to the existing components in place.

| Act | Component(s) | ScrollTrigger effect |
|-----|--------------|----------------------|
| 1. Gap | `Hero` | Scrubbed exit parallax: as the section scrolls past, headline + stat line fade opacity 1→0.3 and translateY 0→-24px, tied to scroll progress (`scrub: true`, `start: "top top"`, `end: "bottom top"`). No entrance animation — hero is visible on load, per the existing `reveal`/`in` convention (elements above the fold start in their settled state, never animate in on first paint). |
| 2. Fix | `TheFix` | One-shot staggered reveal via `ScrollTrigger.batch()`: cards/content blocks fade+translateY(18px→0) in a stagger as the section crosses `start: "top 75%"`, `toggleActions: "play none none none"` (fires once, does not reverse on scroll-up — avoids jank re-triggering). |
| 3. Outcomes | `Outcomes`, `RoiCalculator` | TIME/EFFICIENCY/PROFIT stat numerals count up from 0 to their final value on entry (GSAP number-tween via `innerText`/`textContent` interpolation, `duration: 1.2`, `ease: "power2.out"`), triggered once at `start: "top 70%"`. Calculator container does a simple one-shot fade/scale-in immediately after (stagger offset +0.15s), no scrub. |
| 4. Offer | `Offer` | Asymmetric stagger-in: the widest step-1 card enters first (translateX(-24px)→0 + fade), the two narrower cards follow staggered (translateY(18px)→0 + fade), `start: "top 75%"`, one-shot. |
| 5. CTA | `ProcessTransparency`, `FinalCta` | ProcessTransparency: a connecting progress line/rule fills via `scrub: true` tied to the section's own scroll range as the visitor scrolls through the numbered steps (the one genuinely scrubbed, continuous effect in this section — everything else in this table is a one-shot trigger). FinalCta: simple one-shot scale-in (0.96→1 + fade) on entry, `start: "top 80%"`; the button's own static dual-glow + shine-sweep are pure CSS (see sketch reference), not GSAP-driven — do not reimplement them in GSAP. |

Use `ScrollTrigger.batch()` for any effect touching multiple sibling cards (TheFix, Outcomes, Offer, ProcessTransparency) rather than one `ScrollTrigger.create()` per card — fewer trigger instances, better scroll performance, matches the "budget-conscious solo maintainer" constraint (less code to maintain, cheaper to run).

### Reduced motion (DSGN-03) — required gate on every animation, GSAP and CSS alike

**GSAP layer:** wrap ALL ScrollTrigger registrations in `gsap.matchMedia()`:

```js
const mm = gsap.matchMedia();
mm.add("(prefers-reduced-motion: no-preference)", () => {
  // every ScrollTrigger-driven effect from the table above goes here
});
mm.add("(prefers-reduced-motion: reduce)", () => {
  // set every animated element directly to its final/settled state —
  // opacity: 1, transform: none — no tween, no ScrollTrigger instance created at all
});
```

This is not optional per-effect — a visitor with `prefers-reduced-motion: reduce` must get a fully readable, fully static page with zero ScrollTrigger instances created (not just animations that complete instantly — no scroll-jacking, no pinning, nothing).

**CSS layer (pure-CSS decorative motion, independent of GSAP — glow-box blob rotation, shine-sweep, marquee ticker, magnetic-button follow, status-pill ping):** add a blanket override —

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

Verify this override actually reaches the glow-box `::before`/`::after` pseudo-elements and the ticker's `@keyframes` — test with the OS-level reduced-motion setting on, not just a code review, before marking DSGN-03 done.

### Core Web Vitals budget (DSGN-04) — mobile, with the animation layer active

| Metric | Budget | How this contract protects it |
|--------|--------|-------------------------------|
| LCP < 2.5s | Hero H1 is server-rendered plain text/CSS — zero JS dependency to paint. GSAP is dynamically imported and never blocks initial render. `next/font` self-hosts JetBrains Mono with `display: swap` + automatic fallback-font metric matching (prevents FOIT and reduces the swap-triggered reflow). About page's photo (`annie-photo.jpg`, move from repo root to `public/`) uses `next/image` with explicit `width`/`height` and is NOT the homepage's LCP element (About is a separate route). |
| CLS < 0.1 | All GSAP/CSS animations touch only `transform`/`opacity` — never `width`/`height`/`top`/`left`/layout-affecting properties. Glow-box `::before`/`::after` are `position: absolute`, out of flow — zero layout impact even while animating. `next/image` explicit dimensions everywhere an image ships (About page photo/placeholder). Sticky header/CTA bar heights are fixed, not content-dependent. |
| INP < 200ms | GSAP's ScrollTrigger uses internal `requestAnimationFrame` batching, not raw scroll-event listeners — do not add any additional manual `scroll` event listeners alongside it. `ScrollTrigger.batch()` (not per-card triggers) keeps the number of live trigger instances low. Keep `will-change: transform` scoped to elements actively mid-animation (add via the animation's `onStart`, remove via `onComplete`/`onReverseComplete`) — do not leave it globally set on every card, which forces unnecessary compositor layers and can hurt INP on lower-end mobile devices. |

Manual verification gate for the executor/planner: run a mobile-throttled Lighthouse pass (Chrome DevTools, "Mobile" + "Slow 4G"/"Fast 3G" throttling, or Lighthouse CI) against the built `/` route after the animation layer lands — do not rely on desktop/unthrottled numbers to sign off DSGN-04.
